Chapter 1
How We Approach This Build
1.1

The one idea that makes 4 people × 5 weeks work

Core strategy
Define three contracts by the end of Week 1 — the API surface, the shared graph-access
function signature, and the streaming event schema — and then let all four tracks build against
those contracts independently. Nobody waits for anybody else’s internal implementation to
be finished; everybody waits only for the shape of the interface, which is fixed on Day 5 and
never renegotiated except at a sync point.
If this project fails, it will not fail because someone wrote bad code. It will fail because two people
silently assumed different shapes for the same interface and only discovered it in Week 4. Every
rule in this chapter exists to prevent that one failure mode.

1.2

Five working principles

1. Contracts before code. The request/response shape of every endpoint (Chapter 3) is written
down and agreed before anyone implements the logic behind it. A contract change after Week
1 requires all four people to acknowledge it in the same conversation, not a silent Slack message
to one person.
2. Build against a stub, not a wait. If Track C needs Track B’s retrieval function before it
exists, Track C writes a stub that returns fixture data in the exact shape of the real contract,
and swaps it out later without changing its own code. This is how Weeks 1–2 achieve full
parallelism.
3. Vertical slices over horizontal layers. Each week ends with something that actually runs
end-to-end, however roughly (Chapter 4), rather than four people each perfecting one layer in
isolation until Week 5.
4. Freeze scope, not quality. Week 5 adds zero new features (Section 4.6). Anything not
named in the Ground Source of Truth chapters 3–18 is written into the roadmap-appendix
parking lot (Chapter 8) and not built this cycle.
5. Every stage degrades, nothing crashes. Each track owns building the fallback for its own
stage (Chapter 6) as a first-class deliverable, not a stretch goal bolted on in Week 5.

2.3

Weekly rhythm

• Monday (start of week): 20-minute standup against this document — each track states this
week’s deliverable from Chapter 4 and any dependency it is waiting on.
• Wednesday (midweek check): Async check-in; anyone blocked flags it immediately rather
than at Friday sync, since a 2-day slip on one track’s contract is a 5-week-project risk.
• Friday (sync point): The specific, testable sync-point criterion for that week (Chapter 4)
is demonstrated live by whoever owns it — not described, demonstrated. If it cannot be
demonstrated, it is not done, regardless of how much code exists.

Chapter 8
Parking Lot — What We Are Deliberately Not Building
The rule
If, while building any part of this roadmap, a team member wants to add a service, database,
agent, or framework not already named in the Ground Source of Truth (Ch. 3–18), the correct
action is to write it here and keep building the MVP as scoped — not to build it now.
• Kafka, Kubernetes/K3s, a bitemporal claim store, ontology versioning, a typed Specification
node label.
• A DuckDB/ClickHouse analytical mirror; RT-DETR+SAM2 line tracing for P&IDs.
• Three-tier autonomy with an OPA policy engine; Keycloak/OIDC/mTLS; OT/historian integration.
• Offline-first PWA with CRDT sync; Microsoft GraphRAG-style offline community summarisation; Neo4j GDS Enterprise.
• Fine-tuned models; a published-benchmark-style evaluation with inter-annotator agreement
studies.
• A dedicated material-compatibility lookup database; a company-specific selectionRule relationship type.
• A fourth agent worker; a LiteLLM abstraction layer or a third/fourth model tier.
If the project continues past the hackathon, Ground Source of Truth Ch. 20 (Future Enterprise
Roadmap) gives the trigger condition and recommended sequencing for each of these — that
chapter, not this one, is where post-MVP investment gets planned.
